# مشخصات فنی و رفتاری Wumpus World

**وضعیت:** Draft v0.1 — آمادهٔ بازبینی پیش از پیاده‌سازی  
**دامنه:** نسخهٔ تک‌عامل، گرید ثابت ۸×۸، سه حل‌کننده

## 1. هدف و حدود

سیستم باید یک نقشه و پارامترهای بازی را بخواند، قوانین Wumpus World تعریف‌شده توسط استاد را با یک موتور مشترک اجرا کند، سه عامل متفاوت را روی آن اجرا کند و خروجی قابل‌مقایسه تولید کند.

### داخل دامنه

- parser و validation ورودی؛
- موتور قطعی بازی و تولید ادراک؛
- A\* آفلاین با نقشهٔ کامل؛
- عامل آنلاین قاعده‌محور با مشاهدهٔ ناقص؛
- عامل آنلاین یادگیری نظارت‌شده؛
- تولید داده، آموزش، اجرای batch و گزارش معیارها؛
- CLI، تست و مستندات.

### خارج از دامنهٔ نسخهٔ پایه

- رابط گرافیکی؛
- بازی دونفره؛
- شلیک تیر به غول؛
- حرکت قطری؛
- محیط پویا یا جابه‌جایی غول؛
- یادگیری تقویتی یا الگوریتم ژنتیک؛
- سرویس وب و پایگاه داده.

## 2. واژگان و مدل مشاهده

| واژه | تعریف |
| --- | --- |
| Hidden map | آرایش کامل دیوار، چاه، غول و طلا؛ فقط موتور و عامل جست‌وجو می‌بینند |
| Observation | اطلاعات مجاز عامل آنلاین در یک گام |
| Percept | حس محلی شامل breeze، stench و glitter |
| State | وضعیت کامل بازی شامل موقعیت، جان، طلاهای باقی‌مانده و پایان |
| Knowledge state | باورهای عامل از خانه‌های `SAFE`، `PIT?`، `WUMPUS?`، `UNKNOWN` و بازدیدشده |
| Episode | یک اجرای کامل از شروع تا برد، باخت یا توقف ایمنی |

## 3. قالب ورودی

قالب اصلی پیشنهادی:

```text
********
**D*****
*****G**
W***P***
********
********
********
********
100
25
-10
8 8
```

### قواعد parser

1. هشت خط اول دقیقاً هشت نویسه و فقط از مجموعهٔ `{*, P, W, D, G}` دارند.
2. خط ۹: `initial_health`، عدد صحیح مثبت.
3. خط ۱۰: `gold_value`، عدد صحیح نامنفی.
4. خط ۱۱: `pit_score_delta`، عدد صحیح؛ در مدل داخلی باید نامثبت باشد. برای جلوگیری از خطای ورودی، مقدار مثبت با هشدار به `-abs(value)` نرمال می‌شود.
5. خط ۱۲: خروج؛ parser قالب‌های `row column`، `row,column` و `(row,column)` را می‌پذیرد، ولی مستندات فقط `row column` را تولید می‌کنند.
6. مختصات بیرونی یک‌مبنا و به‌شکل `(row, column)` است. اولین خط نقشه `row=1` و چپ‌ترین نویسه `column=1` است.
7. شروع ثابت `(1,1)` است.
8. شروع و خروج نباید دیوار یا غول باشند. قرارگرفتن چاه در شروع/خروج در نسخهٔ پایه نامعتبر است. اگر شروع طلا داشته باشد، طلا هنگام مقداردهی اولیه مثل ورود به خانه برداشته می‌شود.
9. خطوط خالی انتهایی مجاز، ولی دادهٔ اضافهٔ غیرخالی خطاست.

> نکته: نمونهٔ ناقص داخل متن رسمی یک سطر کوتاه دارد و سپس `...` نشان می‌دهد؛ آن قطعه صرفاً نمایشی است و ورودی معتبر محسوب نمی‌شود.

## 4. مدل داده

```text
Position(row: int, col: int)
Tile = EMPTY | PIT | WUMPUS | WALL | GOLD
Action = UP | DOWN | LEFT | RIGHT
Status = RUNNING | WON | DEAD_WUMPUS | DEAD_HEALTH | NO_SOLUTION | STEP_LIMIT

GameConfig(
  initial_health,
  gold_value,
  pit_score_delta,
  exit_position,
  grid_size=8,
  pit_rounding="floor_with_min_one",
  max_steps
)

GameState(
  position,
  health,
  collected_gold,
  remaining_gold,
  pit_entries,
  steps,
  status,
  event_log
)
```

نماد `G` پس از برداشت در state به خانهٔ خالی تبدیل می‌شود؛ نقشهٔ پایه immutable می‌ماند و `remaining_gold` مشخص می‌کند کدام طلاها هنوز وجود دارند.

## 5. رفتار موتور

### 5.1 کنش قانونی

- مقصد باید داخل گرید و غیر دیوار باشد.
- عامل‌های استاندارد فقط کنش قانونی می‌دهند.
- اگر عامل کنش غیرقانونی داد، runner آن episode را با خطای عامل ثبت می‌کند؛ موتور بی‌صدا آن را به حرکت دیگری تبدیل نمی‌کند.

### 5.2 ترتیب رویداد یک حرکت

برای state در حال اجرا و action قانونی:

1. موقعیت به مقصد تغییر می‌کند.
2. `steps += 1` و `health -= 1`.
3. اگر `health <= 0`، وضعیت `DEAD_HEALTH` و episode تمام می‌شود.
4. اگر مقصد غول است، وضعیت `DEAD_WUMPUS` و episode تمام می‌شود.
5. اگر مقصد چاه است، `pit_entries += 1` و `health = max(1, health // 2)`.
6. اگر مقصد طلای باقی‌مانده است، طلا خودکار برداشته می‌شود.
7. اگر مقصد خروج است و عامل زنده است، وضعیت `WON` می‌شود.
8. در غیر این صورت، observation جدید تولید می‌شود.

این ترتیب یک **قرارداد موقت** است و پس از پاسخ استاد می‌تواند فقط در موتور تغییر کند.

### 5.3 ادراک

در موقعیت فعلی:

- `breeze=True` اگر حداقل یک چاه در چهار همسایگی باشد؛
- `stench=True` اگر حداقل یک غول در چهار همسایگی باشد؛
- `glitter=True` اگر طلای برنداشته‌شده روی خود خانه باشد؛
- `at_exit=True` اگر موقعیت فعلی خروج باشد؛
- عامل موقعیت، جان، طلاهای جمع‌شده، تعداد گام و actionهای قانونی را نیز می‌داند.

ادراک، نوع خانه‌های ناشناخته یا مکان خطرها را افشا نمی‌کند.

### 5.4 برد و امتیاز

- برد: رسیدن زنده به خروج؛ برداشتن همهٔ طلاها در قرارداد فعلی اجباری نیست.
- امتیاز نهایی رسمی فقط برای `WON` گزارش می‌شود:

```text
final_score = remaining_health
            + collected_gold * gold_value
            + pit_entries * pit_score_delta
```

چون `pit_score_delta <= 0` است، جملهٔ آخر جریمه است. برای episode ناموفق، `final_score` تهی (`None`) و یک `diagnostic_score` با همین فرمول صرفاً برای تحلیل ذخیره می‌شود تا شکست با برد اشتباه نشود.

### 5.5 پایان ایمنی

- هر episode یک `max_steps` دارد تا عامل در حلقهٔ بی‌نهایت نماند.
- مقدار پیش‌فرض پیشنهادی: `min(initial_health, 4 * grid_size * grid_size)`؛ چون بدون اثر چاه، جان خودبه‌خود حد بالای واقعی می‌سازد.
- در صورت عبور از حد، وضعیت `STEP_LIMIT` ثبت می‌شود.

## 6. API عامل‌ها

رابط مفهومی مشترک:

```python
class Agent(Protocol):
    def reset(self, config, public_map_info, seed) -> None: ...
    def choose_action(self, observation) -> Action: ...
    def observe_transition(self, observation, action, outcome) -> None: ...
```

- `public_map_info` فقط اندازه، موقعیت شروع و خروج و مقادیر امتیاز را دارد.
- SearchAgent استثنائاً از `FullMapAgent` استفاده می‌کند و نقشهٔ کامل را می‌گیرد؛ این تفاوت در نتایج علامت‌گذاری می‌شود.
- RuleAgent و MLAgent هرگز reference شیء `GameMap` مخفی را دریافت نمی‌کنند.

## 7. مشخصات روش اول: A\* آفلاین

### 7.1 مسئلهٔ جست‌وجو

- state: `(position, health, remaining_gold_mask)`؛
- start: `(1,1)` با جان اولیه و تمام طلاها؛
- successor: هر حرکت قانونی طبق موتور؛ successor مرگ‌آور حذف می‌شود؛
- goal: رسیدن زنده به خروج؛
- نقشه و خطرها کاملاً معلوم‌اند؛ breeze/stench استفاده نمی‌شوند.

### 7.2 تبدیل بیشینه‌سازی امتیاز به کمینه‌سازی هزینه

ثابت زیر را در نظر می‌گیریم:

```text
C = initial_health + total_gold_count * gold_value
```

برای هر مسیر موفق:

```text
loss = (initial_health - remaining_health)
     + pit_entries * abs(pit_score_delta)
     + remaining_gold_count * gold_value

final_score = C - loss
```

پس کمینه‌کردن `loss` معادل بیشینه‌کردن امتیاز است. کاهش واقعی جان و جریمهٔ چاه به transition cost افزوده می‌شود و در خروج، ارزش طلاهای برنداشته‌شده به‌عنوان terminal cost اعمال می‌شود.

### 7.3 heuristic

نسخهٔ مطمئن:

```text
h(n) = ManhattanDistance(current_position, exit)
```

این heuristic حداقل هزینهٔ حرکت تا خروج را تخمین می‌زند و دیوار، چاه و طلای ازدست‌رفته را نادیده می‌گیرد، بنابراین بیش‌برآورد نمی‌کند. برای حالت خروج `h=0` و terminal cost جدا اعمال می‌شود.

### 7.4 خروجی تشخیصی

مسیر، امتیاز پیش‌بینی‌شده، زمان برنامه‌ریزی، تعداد stateهای گسترش‌یافته، بیشینهٔ frontier و علت نبودن راه‌حل ذخیره می‌شود.

## 8. مشخصات روش دوم: عامل قاعده‌محور

### 8.1 دانش نگهداری‌شده

- `Visited(c)` و observation هر خانهٔ بازدیدشده؛
- `Safe(c)`؛
- `PossiblePit(c)` / `ConfirmedPit(c)`؛
- `PossibleWumpus(c)` / `ConfirmedWumpus(c)`؛
- `Blocked(c)` برای مرز و دیوارهای کشف‌شده؛
- `Frontier(c)` برای خانهٔ دیده‌نشدهٔ مجاور ناحیهٔ شناخته‌شده.

### 8.2 قواعد حداقلی

برای خانهٔ فعلی `c` و هر همسایهٔ معتبر `n`:

- اگر در `c` نسیم نیست، `n` چاه نیست.
- اگر در `c` بوی بد نیست، `n` غول نیست.
- اگر `n` نه چاه ممکن/قطعی و نه غول ممکن/قطعی باشد، `n` امن است.
- اگر در `c` نسیم هست، حداقل یکی از همسایه‌های حل‌نشده چاه است؛ همه کاندید می‌شوند، نه قطعی.
- اگر در `c` بوی بد هست، حداقل یکی از همسایه‌های حل‌نشده غول است؛ همه کاندید می‌شوند، نه قطعی.
- اگر پس از حذف خانه‌های ردشده فقط یک کاندید برای یک percept باقی ماند، خطر آن خانه قطعی می‌شود.

### 8.3 سیاست کنش

اولویت‌ها:

1. اگر بازگشت به خروج برای زنده‌ماندن ضروری است، خروج؛
2. حرکت به frontier امنِ بازدیدنشده با بهترین utility تقریبی؛
3. حرکت روی مسیر امن به طلای دیده‌شده یا frontier امن؛
4. بازگشت امن به خروج اگر هدف امن دیگری وجود ندارد؛
5. فقط اگر بدون ریسک هیچ راهی نیست، انتخاب کم‌ریسک‌ترین خانهٔ ناشناخته با tie-breaker قطعی مبتنی بر seed.

تمام factهای استفاده‌شده و قاعدهٔ نتیجه‌دهنده در reasoning trace ثبت می‌شوند.

## 9. مشخصات روش سوم: یادگیری نظارت‌شده

### 9.1 تعریف مسئلهٔ ML

- نوع: classification چهارکلاسه برای `UP/DOWN/LEFT/RIGHT`؛
- ورودی: فقط state مشاهده‌پذیر و حافظهٔ ساخته‌شده از observationهای گذشته؛
- label: کنش خبرهٔ A\* روی state واقعی متناظر؛
- inference: امتیاز/احتمال چهار کنش، سپس mask کنش‌های غیرقانونی و انتخاب بیشترین مقدار.

### 9.2 ویژگی‌های نسخهٔ پایه

- جان نرمال‌شده، نسبت گام مصرف‌شده، تعداد طلای جمع‌شده؛
- بردار نسبی خروج؛
- breeze، stench، glitter و at_exit؛
- چهار action قانونی؛
- برای هر ۶۴ خانه چند channel تخت‌شده: visited، safe، possible_pit، possible_wumpus، blocked، known_gold و relative/current position.

هیچ channel از حقیقت پنهانِ چاه یا غول استفاده نمی‌کند.

### 9.3 تولید نمونه

- نقشه‌ها با seed و قیود اعتبار تولید می‌شوند.
- برای هر state قابل‌مشاهده، خبره با نقشهٔ کامل action بهینه را تعیین می‌کند.
- علاوه بر trajectory اصلی، stateهای متنوع و recoverable برای کاهش bias مسیر خبره نمونه‌برداری می‌شوند.
- داده با `map_id` به train/validation/test تقسیم می‌شود؛ یک نقشه فقط در یک split است.
- نسخهٔ generator، config، seed و feature schema همراه داده ذخیره می‌شود.

### 9.4 مدل‌ها

- baseline: انتخاب کلاس پرتکرار پس از legal mask؛
- مدل توضیح‌پذیر: Decision Tree محدود؛
- مدل اصلی پیشنهادی: Random Forest با وزن‌دهی کلاس‌ها؛
- انتخاب نهایی با validation و معیار macro-F1 به‌همراه rollout win rate انجام می‌شود.

### 9.5 معیارهای ML

- آفلاین: accuracy، macro-F1، per-class recall و confusion matrix؛
- آنلاین: win rate، score، Wumpus death، pit entries و moves روی test maps؛
- نتیجهٔ اصلی، عملکرد آنلاین است؛ accuracy تقلید فقط معیار کمکی است.

## 10. تولید نقشه و مجموعهٔ داده

مولد باید پارامترهای تعداد دیوار/چاه/غول/طلا را بپذیرد و این قیود را رعایت کند:

- شروع و خروج خالی از دیوار و خطر؛
- نمادها overlap ندارند؛
- حداقل یک مسیر هندسی از شروع به خروج بدون عبور از دیوار وجود دارد؛
- برای مجموعهٔ آموزشی اصلی، حداقل یک مسیر موفق با جان داده‌شده وجود دارد؛
- map id از seed و محتوای نقشه ساخته می‌شود؛
- مجموعهٔ تست ثابت و پس از انتخاب مدل دست‌نخورده می‌ماند.

پیشنهاد اندازهٔ اولیه برای توسعه: 100/20/30 نقشه؛ نسخهٔ نهایی در صورت مناسب‌بودن زمان اجرا: 1000/200/300 نقشه.

## 15. معیارهای پذیرش سطح بالا
